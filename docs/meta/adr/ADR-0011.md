# ADR-0011: k6 부하 테스트 + Prometheus Remote Write 시각화 통합

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 4 (1차 측정), Phase 5 (재측정)
- Tags: load-testing, k6, observability

## Context and Problem Statement

Phase 4 부하 테스트(BP2 / TP2)는 비동기화(Phase 3) 효과를 정량 측정하는 사이클이다. Phase 5에서 동일 시나리오 재측정 비교가 학습 가치의 핵심. 도구 선정과 결과 시각화 스택이 두 Phase에 걸쳐 일관되어야 한다.

Forces:
- NestJS 실무 권장 도구 (학습 가치)
- 결과 시각화 = 운영 메트릭(ADR-0009 Prometheus)과 동일 대시보드 가능해야 before/after 비교 효율
- Phase 4·5 사이에 도구 변경 금지 (비교 기준 변화 방지)
- 로컬 Docker Compose 환경
- Git 관리·재현성 보장

## Considered Options

(1) k6 + Prometheus Remote Write (선택)
- 도구: k6 (JavaScript/TypeScript 스크립트)
- 시각화: k6 v0.42+ experimental Remote Write Output → 기존 Prometheus → Grafana 동일 대시보드
- Pros: NestJS 업계 표준, Grafana Labs 공식 지원, JavaScript 친화 (서버 코드와 동일 언어), Git 관리 (코드형), Prometheus 통합으로 운영 메트릭과 cross-link
- Cons: k6 v0.42+ experimental output (정식 안정화 대기), VU(Virtual User) 모델 학습 필요

(2) Gatling
- Pros: 강력한 시나리오 표현
- Cons: Scala 기반 러닝 곡선, JVM 런타임 부담

(3) Artillery
- Pros: JavaScript 기반, 친숙
- Cons: 대규모 시나리오에서 k6 대비 기능 제한, Grafana 통합 약함

(4) JMeter
- Pros: 오래된 표준
- Cons: GUI 기반 → Git 관리/재현성 저해, JVM 부담

(5) k6 Web Dashboard (k6 v0.49+ 내장)
- Pros: 추가 인프라 불필요
- Cons: 부하 테스트 결과만 시각화 — 관측성 스택(ADR-0009)과 분리되어 before/after 비교 시 컨텍스트 전환 비용

(6) k6 Cloud Dashboard
- Pros: SaaS 관리형
- Cons: SaaS 비용 + 클라우드 의존, 학습 프로젝트 out-of-scope

## Decision Outcome

(1) 채택.

도구: k6 (JavaScript/TypeScript 스크립트)
시각화: `k6 run --out experimental-prometheus-rw=http://prometheus:9090/api/v1/write script.js` → Grafana 동일 대시보드 (ADR-0009 LGTM과 통합)

테스트 유형 (Phase 4/5 사이클):
- Baseline (단일 VU 저부하) — 기준 응답 시간
- Load (예상 피크) — p50/p95/p99 latency 분포
- Stress (붕괴점 탐색)
- Spike — Phase 3 안정성 패턴(Circuit Breaker/Bulkhead/Rate Limiter) 1개 이상 적용 시 조건부

시나리오 설계: User Journey 패턴 기반 (예: 로그인 → 목록 조회 → 상세 → 좋아요 → 댓글). 단순 endpoint 타격 지양.

Correlation ID: k6 script가 `X-Correlation-Id: k6-vu{N}-iter{M}-<uuid>` 형식 송신 — 부하 테스트 트래픽과 일반 트래픽 구분 가능 (Loki 라벨로 필터링).

Phase 5 재측정: 동일 스크립트 + 동일 RPS + 동일 Ramp-up. 새 유형 추가 금지 (Spike 신규 편입 예외 1회 허용).

## Consequences

긍정:
- before/after 비교가 단일 Grafana 대시보드에서 가능 (운영 메트릭 + k6 결과 통합)
- User Journey 학습 가치 (단순 endpoint 타격 대비 우위)
- Git 관리 → Phase 4 스크립트가 Phase 5 그대로 재실행 (재현성)
- 관측성 오버헤드 측정 가능 (observability.md §7.3 활성/비활성 비교)

부정:
- k6 v0.42+ experimental output 안정화 추세 의존
- 부하 테스트 시 Prometheus Remote Write가 자체 부하 → 관측 오버헤드 측정 별도 (1회차 관측성 비활성 / 2회차 활성)

중립:
- 구체 시나리오·목표 RPS·SLO 수치는 Phase 4 진입 시점 Problem 재작성에서 확정 (현재 알려진 불확실성 3)
- 클라우드 배포 트리거 시점에는 k6 Cloud 또는 자체 k6 cluster로 자연 확장
