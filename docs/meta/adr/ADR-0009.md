# ADR-0009: Grafana LGTM 스택 (Loki + Promtail + Grafana + Tempo + Prometheus)

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 2 (Loki/Promtail/Grafana/Prometheus), Phase 3 (Tempo 트레이싱)
- Tags: observability, lgtm, monitoring, tracing

## Context and Problem Statement

Phase 2 관측성 가시화(BP4 / TP6)에서 로그·메트릭·트레이스 3대 신호 수집·저장·시각화 스택 선정 필요. Phase 4 부하 테스트 결과는 동일 시각화 플랫폼에서 운영 메트릭과 비교 가능해야 한다.

Forces:
- 로컬 Docker Compose 환경 자원 제약 (CPU/메모리)
- 학습 가치 = 업계 표준 OSS 스택 운용 경험
- k6 부하 테스트 결과의 Prometheus Remote Write 호환 (ADR-0011)
- AntiPattern Blind Faith 해소

## Considered Options

(1) Grafana LGTM 스택 (선택)
- Loki: 로그 (label 카디널리티 제한 < 10)
- Promtail: 로그 수집 agent (stdout pipe)
- Grafana: 통합 대시보드
- Tempo: 분산 트레이스 (OpenTelemetry W3C Trace Context)
- Prometheus: 메트릭 (RED + USE)
- Pros: 단일 벤더(Grafana Labs) 통합, OSS + 자체 호스팅 가능, k6 → Prometheus Remote Write 직결, label 정책 일관 (Loki/Prometheus 동일)
- Cons: 5종 컨테이너 추가 (로컬 자원 부담)

(2) ELK 스택 (Elasticsearch + Logstash + Kibana)
- Pros: 로그 검색 강력
- Cons: Java/JVM 자원 부담 큼, 로컬 환경 무거움, 메트릭은 별도 (Metricbeat) — 통합 불완전

(3) CloudWatch Logs + X-Ray + CloudWatch Metrics
- Pros: 관리형
- Cons: 클라우드 의존, out-of-scope (배포 모델 = 로컬 Docker Compose)

(4) Datadog / New Relic (SaaS)
- Pros: 통합·운영 부담 최소
- Cons: SaaS 비용, 학습 프로젝트 out-of-scope

(5) 파일 로깅만 유지
- Pros: 인프라 추가 없음
- Cons: Phase 4 부하 테스트 시 로그 검색·시각화 불가, BP4 학습 목표 미달성

## Decision Outcome

(1) 채택. 도입 시점:
- Phase 2: Loki + Promtail + Grafana + Prometheus + prom-client + node_exporter / mysqld_exporter / redis_exporter
- Phase 3: + Tempo + OpenTelemetry SDK + kafka_exporter

Loki label 정책: `{service, env, level}` 3개만 (카디널리티 < 10).
Prometheus label 카디널리티 상한: 100 / metric. 금지 라벨: user_id, post_id, event_id, correlation_id, idempotency_key, email 등 (PII + 고카디널리티).

보존: 모든 로그 30일 (Loki `retention_period: 720h`). security.md §7 포렌식 로그 보존과 정합.

## Consequences

긍정:
- 단일 Grafana 대시보드에서 로그·메트릭·트레이스 cross-link
- k6 부하 테스트 결과(ADR-0011)가 동일 Prometheus에 저장되어 운영 메트릭과 직접 비교 (Phase 4/5 before/after)
- Correlation ID(ADR-0008)로 cross-signal 추적 (Loki 로그 → Tempo 트레이스 → Prometheus 메트릭)
- 12-Factor App XI (Logs as event streams) 정합

부정:
- docker-compose.yaml 컨테이너 수 증가 (Phase 2 +4, Phase 3 +2)
- 로컬 자원 부담 (각 컨테이너 메모리·디스크)
- 운영 학습 부담 (5종 도구)

중립:
- 클라우드 배포 트리거 시 LGTM은 자체 호스팅 또는 Grafana Cloud (관리형)으로 자연 전환
- Phase 5 부하 테스트 재측정 시 동일 스택 유지 (비교 기준 변화 방지)
