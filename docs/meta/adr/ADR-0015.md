# ADR-0015: User Aggregate Refactoring Towards Patterns (7단계 점진 마이그레이션)

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 1
- Tags: refactoring, migration, data-design

## Context and Problem Statement

ADR-0003 Identity Separation 채택으로 User Aggregate가 현재 상태(UserAuth.uid가 loginId와 email을 혼재)에서 목표 상태(user / user_auth / user_auth_provider / user_info 4테이블)로 이동 필요. 직접 이동 시 외래키 연쇄 변경(Post / PostLike + Phase 1 신설 Comment / Reply)이 동시 발생하여 다음 리스크:
- E2E 테스트 유실: 중간 검증 지점 없이 한 번에 전체 변경 → 회귀 위험
- 롤백 불가: 부분 실패 시 데이터 복구 어려움
- JWT payload (`sub` 필드) 변경 (uid → userId) — 기존 토큰 전수 무효화 동시 발생

Forces:
- methodology-pattern-selection.md 방향결정 층 (Kerievsky 3방향)
- 학습 프로젝트라도 안전한 리팩토링 전략 학습 가치
- Phase 0에서 활성화된 migrations 인프라 기반 (synchronize:false)
- 단계마다 E2E 통과 보증 가능한 분리

## Considered Options

(1) Refactoring Towards Patterns — 7단계 점진 마이그레이션 (선택)
- 중간 단계: user 테이블 신설 → user_auth 재구성 → user_auth_provider 신설 → user_info 재구성 → 외래키 전파 → Service 레이어 재작성 → JWT payload 전환
- 각 단계마다 E2E 통과
- Pros: 단계별 검증 가능, 부분 실패 시 직전 단계로 롤백, 학습 가치 (Kerievsky 3방향 운용 경험)
- Cons: 단계 수 증가로 작업량 증가, 중간 상태에서 데이터 모델 일관성 임시 깨짐 (관리 필요)

(2) Refactoring To Patterns (직접 이동)
- Pros: 빠름, 단순
- Cons: 외래키 + Service 레이어 동시 변경 → 중간 검증 불가, 테스트 유실 리스크 큼

(3) Refactoring Away From Patterns
- Pros: -
- Cons: 현재 구조가 이미 "없는" 패턴 상태라 "이탈"할 대상 없음 (방향 자체 부적합)

## Decision Outcome

(1) 채택. Phase 1 전체 작업의 핵심 축. 절차 상세는 `docs/solution/phase-1/data-migration.md` 7단계 참조.

핵심 원칙:
- user 테이블 신설 → user_auth / user_auth_provider / user_info 재구성 → 외래키 전파 → Service 레이어 재작성 순서로 단계 분리
- 각 단계마다 E2E 테스트 통과 보증
- 기존 JWT payload(sub 필드)가 uid에서 userId로 변경되는 시점에 기존 토큰 무효화 정책 적용 (사용자 재로그인 필요)

migrations 인프라: Phase 0 #79 활성화된 상태(synchronize:false + migrations 활성) 전제.

## Consequences

긍정:
- 안전한 단계별 검증 (E2E 통과)
- Kerievsky 3방향 운용 경험 (BP3 학습 가치)
- 부분 실패 시 직전 단계 롤백 가능
- Refactoring Towards Patterns 선택 자체가 Type A 블로그 글감

부정:
- Phase 1 작업량 증가 (7단계 분리)
- 중간 상태에서 데이터 모델 임시 일관성 깨짐 (test data·로컬 DB 정리 시 주의)
- 기존 JWT 토큰 전수 무효화 (Phase 1 진입 시 사용자 재로그인 안내)
