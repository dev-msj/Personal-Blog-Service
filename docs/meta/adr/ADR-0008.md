# ADR-0008: Interceptor + AsyncLocalStorage Correlation ID 전파

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 2
- Tags: observability, correlation-id, async-context

## Context and Problem Statement

현 로깅은 Winston에만 결합되어 있고 요청 단위 식별자가 없다 (AntiPattern: Blind Faith). Phase 3 EDA 도입(ADR-0004) 후에는 동기 HTTP 요청 → outbox INSERT → Kafka publish → consumer 처리 → BullMQ enqueue → worker 실행 같은 cross-process 흐름이 자주 발생한다. 단일 비즈니스 트랜잭션을 추적할 수단이 필요하다.

Forces:
- 모든 Service 메소드에 Correlation ID 파라미터 추가는 침투성 과다 (Causes of Redesign 6: Tight coupling 악화)
- Node.js 16+ 네이티브 `AsyncLocalStorage` 제공
- NestJS Global Interceptor 컨벤션 (요청 진입 시 컨텍스트 설정)
- Trace ID와 Correlation ID는 의미가 다름 — 분리 보존 필요

## Considered Options

(1) NestJS Global Interceptor + Node.js AsyncLocalStorage (선택)
- 진입 시 요청 헤더에서 `X-Correlation-Id` 추출 or 서버 생성 (UUID v4) → AsyncLocalStorage에 set
- Winston formatter가 AsyncLocalStorage에서 자동 추출하여 모든 로그에 주입
- 비동기 경로(Kafka/BullMQ)에서는 이벤트 envelope `_meta.correlation_id` + Kafka header `x-correlation-id`로 전달, consumer가 AsyncLocalStorage 복원
- Pros: Service 시그니처 침투 없음, NestJS/Node 표준 컨벤션, 비동기 경로 자연 전파 가능
- Cons: AsyncLocalStorage 미숙지 시 디버깅 난이도

(2) 명시적 파라미터 전달
- 모든 Service 메소드에 `correlationId` 추가
- Pros: 의존 명시
- Cons: Shotgun Surgery 악화 — 기존 Service 시그니처 전면 수정, 새 메소드 추가 시마다 추가

(3) cls-hooked (레거시)
- Pros: AsyncLocalStorage 출시 이전 표준
- Cons: Node 16+ AsyncLocalStorage로 대체됨

(4) Correlation ID = Trace ID (OpenTelemetry trace_id 사용)
- Pros: 단일 식별자
- Cons: 트레이싱 비활성화 시 Correlation ID도 사라짐, 둘은 수명 의미가 다름 (클라이언트가 같은 Correlation ID로 여러 요청 묶기 가능)

## Decision Outcome

(1) 채택. Phase 2 도입.

구현 사양:
- 헤더 이름: `X-Correlation-Id`
- 수신 우선순위: 요청 헤더 → UUID v4 형식 또는 `k6-vu{N}-iter{M}-<uuid>` 형식 검증 통과 시 채택, 아니면 서버 생성
- AsyncLocalStorage key: `correlation_id`
- 응답 헤더 echo: 항상 포함 (security.md §4 PII 분류상 비즈니스 식별자, PII 아님)
- Winston formatter 자동 주입
- 비동기 전파:
  - Outbox INSERT: envelope `_meta.correlation_id` 캡처
  - Kafka publish: message header `x-correlation-id` 미러
  - Kafka consume: header 추출 → AsyncLocalStorage set
  - BullMQ job: data 필드 + worker에서 set

Trace ID와 분리: Correlation ID는 비즈니스 식별자, Trace ID는 W3C Trace Context (OpenTelemetry). 본 프로젝트는 1:1 운용하되 필드 분리.

## Consequences

긍정:
- 모든 로그 / 메트릭 / 트레이스에 Correlation ID 자동 포함
- cross-process 추적 가능 (HTTP → Outbox → Kafka → Consumer → BullMQ → Worker)
- k6 부하 테스트 트래픽 식별 가능 (`k6-vu{N}-iter{M}-<uuid>` 형식) → Loki/Tempo cross-link
- AntiPattern Blind Faith 해소의 primary

부정:
- async.md §4.2 envelope 구조에 `_meta.correlation_id` 추가 필요 (Phase 2 도입 시점에 envelope 변경 = backward compatible 추가 필드)
- AsyncLocalStorage 사용 가이드 학습 비용

중립:
- observability.md §1.4 [확정] 구현 사양 — primary 위치
- application-arch.md §채택 패턴 §Interceptor — 패턴 채택 명시 위치
- 모든 외부 의존(Google OAuth, Redis, Kafka)에 Correlation ID 전파 가능하나 학습 프로젝트는 우선 내부 흐름만
