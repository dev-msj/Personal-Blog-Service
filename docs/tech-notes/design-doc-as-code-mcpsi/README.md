# "문서가 곧 코드"가 되려면 — MCPSI에 권장 9단계 설계 프로세스를 흡수시킨 이야기

> Type B (회고 블로그) / Phase 0 회고
> 관련 산출물: MCPSI 파이프라인, abundant-nibbling-toast 마스터 플랜, docs/solution, docs/implementation, wiki-workflow.png

## 개요

이 글은 AI 에이전트(Claude Code)와 함께 운영하기 위해 자체 설계한 MCPSI 파이프라인이 어떤 영역에서 부족했고, 그 부족함을 권장 9단계 설계 프로세스에 비춰 어떻게 보강했는지 정리한 회고다.

핵심 메시지는 한 줄이다. AI 에이전트로 일관된 품질의 코드를 생산하려면 "설계 문서가 코드의 근거"가 되어야 하고, 그 조건을 충족시키려면 큰 그림 결정만으로는 부족하다. 다이어그램과 인터페이스 명세, 그리고 단계 횡단으로 누적되는 메타 산출물(추적표·ADR·QAS·위협 모델)이 함께 있어야 한다.

본문은 발견 → 진단 → 조사 → 결정의 시간순으로 간다.

## 시작점 — Phase 0를 마치고 Wiki를 docs로 옮기려던 순간

Phase 0가 끝난 후 문서화 정리를 시작했다. 그때까지 프로젝트의 설계는 두 곳에 나뉘어 있었다.

- GitHub Wiki: ERD, API 명세, 워크플로우 다이어그램 같은 상세 산출물
- `docs/`(MCPSI 산출물): Meeting Log, Context, Problem, Solution, Implementation의 단계별 결과물

목표는 단순했다. 현시점 구현 내용에 맞춰 Wiki 내용을 갱신하면서 `docs/`로 이전하고 README를 정리하는 일이었다. 그런데 두 산출물을 나란히 펼치는 순간 문제가 드러났다.

**Wiki에 있던 ERD, API spec, 워크플로우 다이어그램이 MCPSI 설계 문서에는 어디에도 들어 있지 않았다.** Solution 폴더에는 overview, application-arch, data-design, security, observability, async-processing 같은 큰 그림 결정만 있었다. 데이터 모델은 글로 묘사되어 있을 뿐 ERD가 없고, API는 보안 정책 수준에서만 언급되고 명세가 없고, 시간축 위의 메시지 흐름은 어떤 다이어그램에도 들어 있지 않았다.

Wiki가 그동안 그 빈자리를 메워 왔다는 뜻이다. 그리고 그건 단순한 위치 문제가 아니었다.

## 발견한 허점 — 큰 그림만 합의된 설계는 일관된 코드를 만들지 못한다

MCPSI는 처음부터 한 가지 목표를 갖고 있었다. AI 에이전트를 바꿔 가며 구현하든 같은 AI로 여러 번 구현하든, 항상 일관된 로직과 80% 이상 동일한 품질의 코드를 생산하는 것. 이를 위해서는 설계 문서가 코드의 근거가 되어야 한다.

문제는 "근거가 된다"는 말의 무게였다.

MCPSI는 Bounded Context, Aggregate, 적용 패턴 같은 구조적 결정은 명확히 기록한다. 그러나 그 결정을 코드로 옮기는 순간 필요한 디테일 — 어떤 모듈이 어떤 시그니처로 어떤 데이터를 주고받는지, 어떤 상태 전이가 어떤 트리거에서 일어나는지, 어느 시점에 어떤 이벤트가 발행되는지 — 은 이슈를 구현하는 시점에 그때그때 판단되고 있었다.

이게 두 가지 결과를 만들었다.

첫째, 같은 큰 그림 결정이라도 구현자(여기서는 AI 에이전트의 컨텍스트와 판단)에 따라 디테일이 달라진다. 일관성이 무너진다.

둘째, 리뷰 사이클에서 디테일 부재가 드러난다. "왜 이 모듈이 저 모듈을 직접 호출하지?", "이 상태에서 저 상태로 가는 분기가 누락됐다" 같은 지적이 누적된다. 지적은 신규 이슈로 등록되고, 신규 이슈가 늘면 이슈 목록이 복잡해지고, 다음 작업의 의존성이 다시 흐려진다.

그리고 한 가지 더. **다이어그램의 부재는 설계 문서의 가시성 자체를 떨어뜨린다.** 글로 쓰인 데이터 모델은 ERD 한 장보다 훨씬 천천히 머리에 들어온다. AI에게나 사람에게나 그렇다.

## 진단 — "문서가 곧 코드"가 무엇을 요구하는가

이 시점에 한 가지를 분명히 했다. "설계 문서가 코드의 근거"라는 원칙은 그 자체로 측정 가능한 조건으로 풀어야 한다는 것이다.

조건은 세 가지로 정리된다.

1. 코드가 의지하는 모든 결정이 문서 어딘가에 있어야 한다. 결정의 근거(Why)와 트레이드오프까지 포함해서.
2. 코드의 시그니처·흐름·상태 전이를 직접 도출할 수 있을 만큼 상세한 산출물이 있어야 한다. 즉 인터페이스 명세, 시간축 다이어그램, 상태 다이어그램, ERD가 누락 없이 갖춰져야 한다.
3. 변경이 발생했을 때 영향 범위를 추적할 수 있어야 한다. 어느 요구가 어느 결정으로, 어느 모듈로, 어느 테스트와 메트릭으로 이어지는지가 한 곳에서 보여야 한다.

조건 1은 MCPSI가 이미 충족하고 있었다. 조건 2와 3이 문제였다.

여기까지 정리하고 나니 한 가지 의문이 남았다. 이런 문제는 분명히 처음 마주치는 것이 아닐 텐데, 기존 소프트웨어 설계 프로세스들은 이 문제를 어떻게 풀고 있는가.

## 조사 — 권장 9단계 설계 프로세스의 골격

Claude Code와 함께 권장되는 소프트웨어 설계 프로세스를 조사했다. 조사 결과는 한 줄로 압축된다. **기존에 정립된 설계 프로세스들은 "설계 문서가 곧 코드가 될 수 있도록" 이미 여러 단계의 절차와 산출물을 갖추고 있었다.**

조사에서 정리한 9단계 골격은 다음과 같다.

1. Context / Domain Discovery — 누가, 왜, 어떤 제약 안에서. 도메인은 어떻게 나뉘는가
2. Requirements (FR + NFR/QAS) — 무엇을 어느 품질로 동작하는가
3. Solution Strategy — 어떤 큰 결정으로 NFR을 만족시키는가
4. Building Blocks — 시스템이 논리적으로 무엇으로 구성되는가, 책임 경계는 어디인가
5. Deployment — 물리적으로 어디서 어떻게 동작하는가
6. Crosscutting Concepts — 모든 모듈에 일관 적용될 정책
7. Runtime / Behavioral — 시간축으로 어떻게 흐르는가. 4단계 책임 경계와 6단계 정책이 NFR을 실제로 충족하는가
8. Interface / API Spec — 외부와 어떤 계약으로 만나는가
9. 설계 → 실행 인계 게이트 — 테스트 전략, Runbook, 운영 인계

그리고 단계가 아니라 "단계 횡단"으로 누적되는 메타 산출물이 별도로 있었다.

- ADR (Architecture Decision Record) — 임계값 이상의 결정 기록
- Risk Register — 리스크와 잔여도
- QAS 추적표 (NFR 슬라이스) — 정량 NFR이 어떤 정책·검증·메트릭으로 이어지는지
- Traceability Matrix — 요구 → FR/NFR → ADR → 모듈 → API → 테스트 → 메트릭
- Glossary — 도메인 용어집

또 하나 인상적이었던 부분은 **다이어그램을 상류 설계의 진단기로 본다는 관점**이었다. State Diagram을 그릴 수 없으면 Aggregate 정의가 부족하다는 신호, Sequence Diagram의 컴포넌트가 과다하면 모듈 응집도가 낮다는 신호, Saga가 너무 길면 BC 경계 오류라는 신호. 다이어그램이 결과물이 아니라 상류 설계의 정합성을 사후 검증하는 도구로 쓰인다.

## MCPSI와의 대조 — 깊은 영역과 빠진 영역

조사 결과를 손에 들고 MCPSI를 다시 봤다. 의외로 깊은 영역과 분명히 빠진 영역이 갈렸다.

MCPSI가 이미 깊은 영역:
- Meeting Log → Context → Problem → Solution → Implementation의 단계별 게이트 운영
- DDD Strategic(Bounded Context)과 Tactical(Aggregate) 양쪽 채택
- arc42·C4·Event Storming·4+1 View 등 표준의 채택과 매핑
- 워크플로우 단계 운영(Phase B의 B1~B16, 리뷰 사이클의 분류 4종)에서 권장 프로세스를 초과하는 메커니즘

빠진 영역은 명확히 6개로 정리되었다.

1. NFR을 Quality Attribute Scenario 6요소(Source/Stimulus/Environment/Artifact/Response/Response Measure)로 정량화하고 Traceability Matrix에 NFR 슬라이스로 추적하는 메커니즘
2. Runtime/Behavioral을 별도 산출물로 분리해 시간축 메시지 흐름·상태 전이·Saga를 시각화하는 산출물
3. ADR × 변경 임팩트 매트릭스 — 결정의 무게(ADR 작성 임계값)와 변경의 영향 범위(국소/단계내/단계간)를 직교로 다루는 운영 규칙
4. 다이어그램 진단 신호 — 비대 신호 9종, 미작성 신호 5종을 단계간 재통과 트리거와 매핑
5. STRIDE 기반 위협 모델링 — Spoofing/Tampering/Repudiation/Information Disclosure/DoS/Elevation of Privilege 6범주
6. Test Pyramid 7종(단위/통합/계약/E2E/성능/보안/카오스)의 비율을 입력 단계 산출물에서 도출하고 동기화

이 6개는 우연한 누락이 아니었다. 다섯은 모두 같은 문제의 다른 표면이다. 즉 **결정의 흔적과 검증의 흔적이 단계 횡단으로 누적되지 않으면, 변경이 들어왔을 때 영향 범위를 추적할 수 없다.** 그러면 "문서가 코드의 근거"라는 원칙이 시간이 갈수록 침식된다.

## 보강한 6개 영역

조사된 골격에 기반해 6개 영역을 보강했다. 각각의 의도와 근거는 다음과 같다.

### QAS 6요소와 Traceability Matrix

정량화되지 않은 NFR은 검증할 수 없고, 검증할 수 없는 결정은 코드의 근거가 되지 못한다. QAS 6요소는 NFR 하나하나를 측정 가능한 시나리오로 풀어 쓰게 한다. 그리고 그 QAS가 어느 ADR로 만족되고, 어느 정책·런타임 검증·테스트 케이스·운영 메트릭으로 이어지는지를 Traceability Matrix가 한 곳에서 잇는다.

이 추적이 있으면 변경이 들어왔을 때 "어디까지 영향이 가는가"를 추측이 아니라 조회로 답할 수 있다.

### Runtime/Behavioral 분리

기존 MCPSI에서 시간축의 메시지 흐름과 상태 전이는 명시적 산출물이 아니었다. 큰 그림 결정과 정책은 있되, "그래서 실제로 어떤 순서로 어떤 컴포넌트가 어떤 메시지를 주고받는가"는 코드를 봐야 알 수 있었다.

Runtime을 별도 영역으로 분리하면 Sequence Diagram, State Diagram, Saga Diagram이 명시적 산출물이 된다. 그리고 부수 효과가 하나 더 있다. 이 다이어그램들이 그려지지 않거나 비대해지는 순간이 곧 상류 설계의 결함 신호다. 진단기로서의 역할이 자동으로 따라온다.

### ADR × 변경 임팩트 매트릭스

결정의 무게와 변경의 영향 범위는 다른 축이다. 단일 모듈만 영향을 주는 변경(국소)이라도 되돌리기 비용이 1주 이상이면 ADR을 작성해야 하고, 다수 단계에 영향을 주는 변경(단계간)이라도 결정이 자명하면 ADR 없이 Meeting Log와 Traceability 갱신만으로 충분하다.

두 축을 직교로 다루는 매트릭스를 도입하면 "이번 변경은 ADR을 써야 하는가, 어느 단계까지 재통과해야 하는가"가 자의적 판단이 아니라 표 조회로 결정된다.

### 다이어그램 진단 신호

다이어그램이 비대해지면 도구를 바꾸기 전에 상류 설계를 점검해야 한다는 관점을 명시적 절차로 만들었다. Sequence가 너무 많은 컴포넌트를 거치면 4단계 모듈 응집도 신호, State가 폭발하면 Aggregate가 두 책임을 가졌다는 신호, Saga가 너무 길면 BC 경계 오류 신호.

각 신호에 단계내/단계간 분기 기준을 매핑하면 발화 즉시 재통과 시작 단계가 자동 결정된다.

### STRIDE 위협 모델

보안은 정책 수준에서만 다뤄지고 있었지만, 무엇을 어디서 어떻게 공격받을 수 있는가에 대한 체계적 분석은 없었다. STRIDE 6범주로 자산과 공격면을 분류하면, 보안 정책과 보안 테스트 케이스가 위협 모델로 정당화된다. "왜 이 인증 정책을 두는가"가 추상적 모범 사례가 아니라 식별된 위협에 대한 응답이 된다.

### Test Pyramid 7종 비율

테스트 비율은 자율 출발점이 아니어야 한다. 단위 테스트 비율은 4단계 모듈 책임 정의에서, 통합은 모듈 간 의존에서, 계약은 8단계 API 명세에서, E2E는 핵심 Use Case에서, 성능은 성능 QAS와 용량 산정에서, 보안은 위협 모델에서, 카오스는 가용성 QAS와 실패 시나리오에서 도출된다.

이 매핑이 있으면 9단계 인계 게이트에서 비율 동기화가 검증 가능한 기준이 된다.

## 추가로 함께 한 결정 — Phase scoping과 이슈별 흐름 다이어그램

조사된 9단계 흡수와 별개로, 운영 효율을 위한 두 가지 결정을 함께 도입했다.

첫 번째는 Solution의 Phase scoping이다. Solution을 공통 골격(common)과 Phase별 증분(phase-{N})으로 분리한다. Aggregate, Bounded Context, 채택 패턴 같은 구조적 결정은 Phase에 종속되지 않으므로 common에 둔다. 배포, 외부 통합, 마이그레이션, 규모 증분 같은 항목은 Phase 단위로 자연스럽게 증가하므로 phase에 둔다. /mcpsi-solution을 호출할 때 대상 Phase의 증분만 생성하면 호출 단위의 집중도가 올라간다.

두 번째는 Implementation 단계의 이슈별 흐름 다이어그램화다. `docs/implementation/flows/{flow-id}.md`에 Mermaid sequenceDiagram으로 처리 흐름을 두고, 이슈 본문에 flow-id로 참조한다. dedup 기준은 "처리 단계 시퀀스 + 분기 구조의 동일성"이다. 예를 들어 생성/수정처럼 SQL의 INSERT/UPDATE 차이밖에 없으면 같은 flow를 공유한다. 이 발상의 근거는 기존 Wiki에 있던 워크플로우 다이어그램의 dedup 패턴("생성/수정", "조회 Comment/Reply")이었다.

이슈와 flow의 다대일 관계가 성립하면 한 가지 효과가 따라온다. **구현이 다이어그램에서 직접 도출되고, 구현자 간 변동성이 줄어든다.** "문서가 곧 코드"가 큰 그림이 아니라 이슈 단위에서 작동하기 시작한다.

## 남은 한계 — 보강 후에도 자동화로 메울 수 없는 영역

보강 자체는 8개 세션(S-0 ~ S-8)에 걸쳐 적용을 마쳤다. 그러나 보강이 끝났다고 모든 영역이 자동화로 풀린 것은 아니다. 보강 방식 자체에 남는 본질적 한계가 두 가지다.

첫째, 이슈별 흐름 다이어그램의 dedup 판별이 형식 검증으로 환원되지 않는다. 도입한 dedup 기준은 "처리 단계 시퀀스 + 분기 구조의 동일성"인데, 이 동일성은 의미적 판단을 요구한다. verify 게이트가 자동으로 잡을 수 있는 부분은 flow-id 누락, 미사용 flow 같은 형식적 항목에 그치고, "이 두 flow가 정말 같은 흐름인가"는 작성자의 판단으로 남는다. 자동화 가능한 영역과 그렇지 않은 영역의 경계를 명확히 두는 것이 운영의 출발점이다.

둘째, Test Pyramid 7종 비율 동기화는 입력 단계 산출물의 정량성에 의존한다. 단위·통합·계약·E2E는 모듈 책임과 API 명세에서 도출되므로 정량화 부담이 작지만, 성능과 카오스 비율은 QAS Response Measure의 정량화 정확도에 직결된다. Problem 단계에서 QAS 정량화가 느슨하게 끝나면 비율 도출 자체가 의미를 잃는다. 이 의존 관계가 보강 효과의 임계점이다.

앞의 한계를 다루는 운영 방식은 두 갈래다.

dedup 판별의 의미적 영역은 자동화로 메우지 않는다. 대신 dedup 결정이 일어날 때마다 결정 근거를 flow 파일 안에 "endpoint variants" 메모로 누적한다. 같은 flow를 공유하기로 한 변종은 어떤 차이 때문에 같은 흐름으로 봤는지가 기록되고, 다른 flow로 분리한 변종은 어떤 차이 때문에 분리했는지가 기록된다. 쌓인 결정 이력이 차후 판단의 참조 사례가 된다. verify는 형식적 검증에 머무르되, 의미적 판단의 일관성은 이 누적된 이력으로 확보한다.

QAS 정량화 정확도는 Problem 단계의 종료 기준을 강하게 잡는 것으로 다룬다. nfr-qas의 모든 QAS는 Response Measure가 측정 가능한 수치로 표현되어야 통과시키고, "측정 가능"의 기준에 "어떤 도구로, 어디서, 어떤 조건에서 측정하는가"가 함께 명시되어야 한다. 정량화가 어려운 QAS는 통과시키지 않고 보강 또는 분리한다. 그래야 Test Pyramid 비율 도출이 의미를 가진다.

이 두 한계와 운영 방식은 보강 방향 자체의 정당성을 흔들지 않는다. 정작 보강의 정당성은 다음 단계에서 답해야 한다.

## 다음 단계 — 검증의 출발점

보강 자체는 적용이 끝났다. 그러나 진짜 검증은 갱신된 MCPSI가 다음 Phase의 작업에 실제로 작동할 때 시작된다.

검증 축은 두 가지다.

1. 갱신된 MCPSI를 통해 작성될 문서들이 실제로 코드의 근거가 될 수 있는 수준에 도달하는가. 즉 다음 Phase의 구현에서 디테일이 이슈 시점이 아니라 설계 시점에 결정되는가.
2. 그 결과로 리뷰 사이클의 신규 등록 이슈가 줄어드는가. 이슈 목록이 클린하게 유지되는가.

두 질문 모두 답이 긍정이라면, 그다음 단계는 워크플로우 자체의 간소화다. 기존 워크플로우는 MCPSI 설계 문서가 부족하다는 전제 위에서 만들어졌기 때문에 과도하게 많은 검증·리뷰 절차를 갖고 있다. 워크플로우 1회 실행에 약 1시간과 5시간 세션 사용량의 25%가 소요되는 상황이다. 이 비용이 정당화되던 전제가 사라지면 워크플로우는 가벼워져도 된다.

워크플로우가 가벼워지면 다시 그다음 단계가 열린다. 이슈 병렬 처리를 검토할 만한 세션 여유가 생기고, 종국에는 설계 문서만 작성하면 사람이 관여하지 않아도 자동으로 모든 이슈가 처리되는 고도화로 향할 수 있다.

여기까지 가는 데 필요한 첫 조건이 "설계 문서가 정말 코드의 근거가 되는가"였다. 이번 보강은 그 조건을 충족시키기 위한 작업이었고, 정말 충족되었는지는 다음 Phase의 작업 결과가 답할 것이다.

## 추신 — Traceability Matrix 운영 중단 (2026-06-08)

위 보강에서 도입한 메타 산출물 중 Traceability Matrix는, 이후 거버넌스 정책 재평가로 본 프로젝트에서의 운영을 중단했다. 9컬럼 매트릭스의 단계별 전 컬럼 동기화 비용이 1인 학습 프로젝트에서 추적 가치를 초과했고, 매트릭스가 제공하던 결정 근거 추적은 ADR 라이프사이클이, 변경 영향 추적은 Change Impact Log가 이미 전담하고 있어 중복 뷰였기 때문이다. ADR 라이프사이클과 Change Impact Log는 잔존한다. 폐지는 본 프로젝트의 운영 차원이며, 방법론(mcpsi-traceability) 차원에서 매트릭스 개념이 제거된 것은 아니다. 근거는 docs/meeting-logs/2026-06-08.md.
