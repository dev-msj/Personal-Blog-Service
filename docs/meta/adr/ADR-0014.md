# ADR-0014: RFC 9457 Problem Details 응답 표준 전환

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 5 (TP8)
- Tags: api, error-handling, http, standard

## Context and Problem Statement

현 구현은 "HTTP 200 + body 내 ErrorCode 5자리" 컨벤션 (자체 표준). 결과:
- 표준 HTTP semantic 미사용 (4xx/5xx 활용 부재) → Smell: Oddball Solution
- 업계 도구(monitoring·proxy·client SDK)와 통합 시 추가 변환 필요 → AntiPattern: Reinvent the Wheel
- 학습 가치 측면에서 RFC 9457 표준 운용 경험 부재

Forces:
- TP8 / BP6 프로덕션 품질 개선
- Spring Boot 3+ / ASP.NET Core 7+ native opt-in 도입 → 사실상 표준 추세
- Phase 5 NestJS 11 + @nestjs/swagger 11 메이저 업그레이드와 응답 인프라 일괄 변경 효율
- 클라이언트 파싱 분기 부담 vs 표준 학습 가치

## Considered Options

(1) Phase 5 일괄 전환 (선택)
- Phase 1~4까지 기존 컨벤션 유지, Phase 5 진입 시 RFC 9457 + 표준 HTTP status code로 일괄 전환
- Pros: 응답 인프라 일괄 변경 (메이저 업그레이드 시점과 동기화), 클라이언트 파싱 분기 1회 도입, 표준 학습 가치 명확
- Cons: Phase 1~4 동안 자체 컨벤션 운영 부담 지속, Phase 5 작업량 집중

(2) 현 컨벤션 영구 유지
- Pros: 변경 없음
- Cons: Smell + AntiPattern 지속, BP6 학습 목표 미달성, 클라우드 배포 시 외부 도구 통합 비용

(3) Phase 1 신규 엔드포인트(Comment/Reply)부터 점진 전환
- Pros: 점진 도입
- Cons: 기존/신규 이중 응답 표준 운영, 클라이언트 파싱 분기 부담 분산되어 영구화, 응답 인프라 두 번 손대는 비효율

(4) RFC 9457 native NestJS 라이브러리 (`@sjfrhafe/nest-problem-details`) 사용
- Pros: 라이브러리 위임
- Cons: Phase 5 진입 시점에 라이브러리 안정성·유지보수 상태 평가 후 결정 — 별도 [가이드] 결정

## Decision Outcome

(1) 채택. Phase 5 진입 시 전환.

Phase 5 응답 형식:

```http
HTTP/1.1 401 Unauthorized
Content-Type: application/problem+json

{
  "type": "https://blog.example/errors/auth-invalid-password",
  "title": "Invalid password",
  "status": 401,
  "detail": "비밀번호가 올바르지 않습니다",
  "instance": "/users/auth/login",
  "errorCode": "AUTH_INVALID_PASSWORD",
  "correlationId": "550e8400-..."
}
```

ErrorCode → HTTP status 매핑 (observability.md §6.2 표):
- Auth (10xxx) → 401/403
- User NOT_FOUND → 404 / ALREADY_EXISTS → 409
- Post/PostLike → 404/409
- Validation (91xxx) → 400
- COMMON_TOO_MANY_REQUESTS → 429
- COMMON_INTERNAL_ERROR → 500
- COMMON_SERVICE_UNAVAILABLE → 503

Phase 1~4 운영:
- 기존 컨벤션 유지
- 예외: Rate Limit 429 (security.md §5 Phase 1 도입) — @nestjs/throttler가 RFC 6585 준수, FailureResponse body에 `code: COMMON_TOO_MANY_REQUESTS` 신설. Phase 5 전환 시 자연 통합

## Consequences

긍정:
- RFC 9457 표준 운용 경험 (BP6)
- 외부 도구(모니터링·proxy·client SDK) 통합 비용 감소
- application/problem+json Content-Type으로 일반 응답과 자연 구분

부정:
- Phase 5 작업량 집중 (모든 ExceptionFilter 재작성 + BaseException에 `httpStatus` 추상 getter + 모든 E2E 응답 검증 로직 재작성)
- 클라이언트 SDK 동시 변경 필요 (HTTP 200 단일 가정 → status code별 분기)
- Swagger 문서 전수 갱신 (`@ApiResponse({ status: 401, type: ProblemDetailsDto })`)

중립:
- observability.md §6.2 [확정] — primary 위치
- application-arch.md Phase 5 행 "Swagger 문서 품질 개선" 범위에 RFC 9457 응답 스키마 추가
- security.md §5 Rate Limit 429 — Phase 5 전환 후 Problem Details 포맷 자연 통합
