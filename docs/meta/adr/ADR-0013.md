# ADR-0013: AES-GCM PK 암호화 전환

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 5 (TP8)
- Tags: security, encryption, migration

## Context and Problem Statement

현 구현은 응답 PK를 `crypto-js` AES-ECB 모드로 암호화 (PK_SECRET_KEY 16자 = 128bit). ECB 모드는 동일 평문 → 동일 암호문 → 패턴 노출 취약 (Smell: Oddball Solution). 학습 가치 측면에서 OWASP 권장 인증 암호화(Authenticated Encryption)로 전환 필요.

Forces:
- TP8 / BP6 프로덕션 품질 개선
- OWASP Cryptographic Storage Cheat Sheet GCM 또는 CBC-HMAC 권장
- 기존 암호문 보존 — Expand-and-Contract 마이그레이션
- Phase 4 baseline 보호

## Considered Options

(1) AES-256-GCM (선택)
- 키 길이: 256bit (`PK_SECRET_KEY` 32자로 확장)
- IV: 12바이트 per-encryption 랜덤 (`crypto.randomBytes(12)`)
- Authentication Tag: 16바이트
- 포맷: `base64url(iv || ciphertext || tag)` — 단일 URL-safe 문자열
- Pros: Authenticated Encryption 내장 (단일 API), NIST SP 800-38D 표준, Node.js `crypto` 표준 라이브러리 지원, IV 랜덤화로 패턴 노출 차단
- Cons: 포맷 변경으로 응답 길이 증가 (12B IV + 16B tag = 28B 추가)

(2) AES-256-CBC + HMAC-SHA256
- Pros: 동일 알고리즘 그룹
- Cons: HMAC 별도 계산·검증 필요 (복잡도 증가), GCM 단일 API가 운영 단순성 우위

(3) XChaCha20-Poly1305
- Pros: 성능 우위, IV 길이 24B로 충돌 안전성 우수
- Cons: Node.js `crypto` 표준 미포함 (`sodium` 외부 라이브러리 필요)

(4) AES-ECB 유지
- Pros: 변경 없음
- Cons: 패턴 노출 취약 (Smell 지속), Reinvent the Wheel AntiPattern

## Decision Outcome

(1) 채택. Phase 5 진입 시 전환.

Expand-and-Contract:
- `PK_SECRET_KEY` 환경변수 교체 (32자, AES-256). 기존 키는 `PK_SECRET_KEY_LEGACY`로 보존 (구버전 복호화 전용)
- `DecryptPrimaryKeyPipe`: 두 포맷 지원 (Phase 5 과도기). base64url decode 후 길이 판별로 신/구 분기
- `EncryptPrimaryKeyInterceptor`: 신규 포맷만 생성 → 모든 응답이 신버전으로 자연 전환
- Drain 완료(모든 클라이언트가 신버전 PK로 재요청) 후 구버전 복호화 로직 + `PK_SECRET_KEY_LEGACY` 제거

## Consequences

긍정:
- 패턴 노출 차단 (IV 랜덤화)
- Authenticated Encryption — 암호문 변조 감지 가능
- STRIDE-8 (Information Disclosure: PK 패턴 노출) 완화

부정:
- 응답 PK 컬럼 길이 증가 (~28B + base64url 변환 오버헤드 ~37% — 클라이언트 파싱 부담 미미)
- Phase 5 과도기 동안 두 포맷 복호화 로직 공존 (Drain 후 단순화)
- Phase 4 baseline 대비 PK 직렬화 시간 변화 (Phase 5 재측정의 명시적 변수)
