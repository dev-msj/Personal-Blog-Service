# ADR-0005: Transactional Outbox

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 3
- Tags: async, reliability, outbox

## Context and Problem Statement

ADR-0004 EDA 채택으로 도메인 변경 후 이벤트 발행 필요. 단순 구현(commit 후 직접 Kafka publish)은 다음 실패 모드를 안는다:
- DB commit 성공 + Kafka publish 실패 → 이벤트 영구 누락. 복구 불가능
- DB commit 실패 + Kafka publish 성공 → 가짜 이벤트 발행. consumer 상태 오염

at-least-once 의미론을 충족하려면 "DB 변경"과 "이벤트 발행"의 원자성을 보장해야 한다.

Forces:
- 학습 프로젝트라도 운영 가능한 신뢰성 패턴 학습 (BP1)
- 인프라 복잡도 vs 신뢰성
- 단순성 (Helland "Idempotence" + Richardson "Microservices Patterns" 권장)

## Considered Options

(1) Transactional Outbox (선택)
- 구조: 도메인 변경과 같은 트랜잭션 내에 `outbox` 테이블 INSERT. 별도 relay worker가 polling으로 outbox → Kafka publish 후 `published_at` UPDATE
- Pros: DB 원자성 활용한 발행 보장 (at-least-once), 학습 가치 명확 (EIP 표준 패턴)
- Cons: relay worker 별도 운영, outbox 적체 모니터링 필요 (observability §2.2 `outbox_pending_count` Gauge)

(2) Direct publish (commit 후 직접 Kafka 발행)
- Pros: 인프라 단순 (outbox 테이블 불필요)
- Cons: 위 이벤트 누락 위험. 복구 불가. 신뢰성 패턴 학습 가치 부재

(3) Debezium CDC (MySQL binlog 읽기)
- Pros: 도메인 코드에 outbox INSERT 불필요 (CDC가 binlog → Kafka 자동)
- Cons: 인프라 복잡도 큼 (Debezium Connect 클러스터 추가), 학습 프로젝트 스코프 대비 과다. 후속 확장 여지로 기록

(4) Two-Phase Commit (XA)
- Pros: 강한 원자성
- Cons: Kafka XA 미지원 (사실상 불가), 성능·운영 부담

## Decision Outcome

(1) 채택. outbox 테이블 신설 (canonical DDL은 data-design.md §outbox primary).

핵심 컬럼 구조:
- `outbox_id BIGINT AUTO_INCREMENT PK` — relay polling 순서 키
- `event_id CHAR(36) UNIQUE` — UUID v4, 이벤트 envelope 멱등 키
- `aggregate_type, aggregate_id, sequence_number` — Aggregate별 단조 증가 순서 (`UNIQUE (aggregate_type, aggregate_id, sequence_number)`)
- `event_type, schema_version, occurred_at, payload JSON` — envelope 본체
- `published_at NULL` — Relay Worker가 발행 후 set
- `publish_attempts INT` — 재시도 카운터
- `INDEX (published_at, outbox_id)` — polling 쿼리 `WHERE published_at IS NULL ORDER BY outbox_id LIMIT 100` 최적화

Relay worker: NestJS schedule 기반 polling (1초 간격) 또는 독립 워커. polling → Kafka producer.send → `UPDATE outbox SET published_at = NOW() WHERE outbox_id IN (...)`.

## Consequences

긍정:
- at-least-once 보장 (consumer Idempotency와 결합하여 effective exactly-once)
- 적체 모니터링 가능 (`published_at IS NULL` 건수 = `outbox_pending_count` Gauge)
- Phase 4 부하 테스트 시 outbox latency 분리 측정 (`outbox_publish_lag_seconds` Histogram)

부정:
- relay worker 별도 process 관리 (헬스체크·자동 재시작)
- DB write 부하 증가 (outbox INSERT 1회 추가 per 도메인 변경)
- polling 간격 vs publish lag 트레이드오프 (1초 → p99 publish lag ≤ 2초 수준)
