# ADR-0003: Identity Separation + Account Linking

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 1 (TP5)
- Tags: domain, identity, oauth, refactoring

## Context and Problem Statement

현 User Aggregate는 `uid` 필드가 일반 가입 사용자에게는 loginId, OAuth 사용자에게는 email을 의미하는 식별자 혼재 구조다 (Causes of Redesign 4: object representation의 의미 혼재, Smell: Oddball Solution). 결과:
- 같은 사람이 일반 가입 후 Google OAuth로 로그인하면 다른 User로 분리 생성됨 (동일인 식별 불가)
- 외래키(`post.userUid`, `post_like.userUid` 등)의 의미가 사용자마다 다름
- email 변경 이벤트 시 외래키 연쇄 갱신 필요 (전파 비용)

Forces:
- 이상적 CI (휴대전화 인증·실명 인증)는 예산 제약으로 불가 (context.md [예산])
- 학습 프로젝트라도 BP3 (기능 완성도) 학습 목표에서 동일인 식별·계정 연동 경험은 의미 있음
- 외부 IdP는 OAuth `sub`(provider_subject)가 영구 고유 식별자 표준 — 이메일 기반 식별은 변경 가능성·탈취 리스크 존재

## Considered Options

(1) Identity Separation + Account Linking (선택)
- 구조: 내부 영구 식별자 `userId BIGINT` 도입 + `user_auth` (로컬 자격) + `user_auth_provider` (OAuth 자격) + `user_info` (프로필) 4테이블 분리
- 동일인 식별: OAuth 로그인 시 payload.email로 기존 user 탐지 후 자동 연결 (학습 프로젝트 단순화)
- Pros: 동일인 식별 가능, 외래키 의미 통일(`userId`), 추후 IdP 추가 시 user_auth_provider 1행 추가만으로 확장
- Cons: 자동 연결의 탈취 리스크 (이메일 변경·재발급 시) — Phase 5 OAuth `sub` 검증 강화 후속 과제로 명시

(2) 현 구조 유지 + 중복 가입 수용
- Pros: 변경 없음
- Cons: 동일인 식별 불가, BP3 학습 목표 일부 미달성

(3) userId 도입 + uid 유지 (email 기반 연동 없음)
- Pros: 식별자 분리는 됨
- Cons: 동일인 식별 목적 미충족, 외래키 의미는 통일되나 핵심 학습 가치 누락

(4) 명시적 계정 연결 UX (자동 연결 대신 "Google 연결" 버튼)
- Pros: 실무 보안 기본 (탈취 방지)
- Cons: UI 흐름 추가 필요, 자동 연결 방식의 탈취 리스크는 후속 과제로 편입 가능 (현 단계 학습 가치 우선)

(5) 유료 본인인증 CI API
- Pros: 가장 안전한 동일인 식별
- Cons: 예산 제약 (context.md [예산]), 학습 프로젝트 out-of-scope

## Decision Outcome

(1) 채택. Phase 1에서 4테이블 구조로 재설계:
- `user (user_id BIGINT PK, ...)` — Aggregate Root
- `user_auth (user_id, login_id UNIQUE, password, ...)` — 로컬 가입 자격
- `user_auth_provider (user_id, provider, provider_subject, email, ..., UNIQUE(provider, provider_subject))` — OAuth 자격
- `user_info (user_id, nickname UNIQUE, introduce, ...)`

마이그레이션 전략: Refactoring Towards Patterns (ADR-0015 참조). 직접 이동이 아닌 7단계 점진 마이그레이션으로 외래키 연쇄 변경 리스크 분산.

OAuth 자동 연결: payload.email 기반 (학습 프로젝트 범위). Phase 5 OAuth `sub` 검증 강화는 별도 과제.

## Consequences

긍정:
- 동일인 식별 가능 — 일반 가입 + OAuth 같은 email이면 같은 User
- 외래키 통일 (`userId`) — Post / PostLike / Comment / Reply 모두 `user_id` 참조
- 신규 IdP 추가 비용 낮음 (user_auth_provider 1행)
- STRIDE-2 (OAuth 위조) 완화 — provider, provider_subject UNIQUE 제약

부정:
- 외래키 마이그레이션 비용 (Phase 1 data-migration.md 7단계 절차)
- JWT payload의 `sub` 필드 의미 변경 (uid → userId) — 기존 토큰 전수 무효화 필요
- 이메일 기반 자동 연결의 탈취 리스크 — 후속 과제로 기록

중립:
- security.md §4 PII 분류: user_auth_provider.email은 직접 식별자, provider_subject는 준식별자
- Phase 5 검토: 이메일 변경 이벤트 처리 + 수동 연결 UX 편입
