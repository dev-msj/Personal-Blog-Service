# ADR-0001: 모듈러 모놀리스 단일 BC "Blog Service"

- Status: Accepted
- Date: 2026-05-12
- Phase: 전 Phase 공통 (Phase 0~5)
- Tags: architecture, bounded-context, ddd

## Context and Problem Statement

Personal Blog Service는 개인 학습/연구 목적의 NestJS 10 블로그 API 서버다. 비동기 시스템 적용·부하 테스트 학습이 목표지만 도메인 자체는 단순(Post + User 2개 Aggregate). 분산 구조(MSA·다중 BC) 학습을 위해 도메인을 인위적으로 쪼개야 하는가, 혹은 단일 BC로 유지하는가의 의사결정이 필요하다.

Forces:
- 학습 목표는 비동기 시스템 + 부하 테스트이지 MSA 분해 자체가 아님
- 단일 개발자 + 로컬 Docker Compose POC 배포 모델
- 도메인 경계(BC) 분리 필요성을 정당화할 외부 BC 통합·이종 도메인 미존재
- Google OAuth 1건만 외부 서비스 — 통합 수준 ACL 흡수로 충분

## Considered Options

(1) 단일 BC "Blog Service" 모듈러 모놀리스 (선택)
- Pros: Vernon Rule 1 (Aggregate 내부 강일관성)을 트랜잭션으로 직접 보장, 운영 단순성, 학습 목표(비동기·부하 테스트)에 자원 집중
- Cons: MSA 분해 학습 기회 없음 (학습 목표 외이므로 무관)

(2) Auth BC + Content BC 분리
- Pros: BC 분리 학습 가능
- Cons: 도메인 왜곡 (User-Post 결합도가 자연스러운 도메인을 인위 분리), Context Map 패턴이 학습 목적 외 복잡도 도입, 동일인 식별·계정 연동(Phase 1 TP5)이 BC 경계 넘어가는 분산 트랜잭션화

(3) MSA (User Service + Blog Service + Auth Service)
- Pros: 분산 시스템 학습 가능
- Cons: 로컬 POC 환경에 과다, BP1 (비동기 시스템 적용) 학습 목표와 직교, 도메인 규모 불일치

## Decision Outcome

단일 BC "Blog Service" 모듈러 모놀리스 채택. NestJS feature 모듈(`auth/`, `user/`, `blog/`, 향후 `observability/`, `notification/`)이 BC 내부 sub-module로 구성된다. Context Map 9개 관계 패턴(Shared Kernel·Customer/Supplier·Conformist·ACL·OHS·Published Language·Separate Ways·Partnership·Big Ball of Mud)은 모두 해당 없음.

근거: methodology-ddd.md Context Mapping (Evans 2003 / Vernon 2013) — BC 분리 정당화 사유 부재. 외부 통합인 Google OAuth는 BC 간 관계가 아닌 외부 서비스 ACL 수준 흡수.

## Consequences

긍정:
- application-arch.md §모듈 구조 (최종 형상) 정의: User·Post 2개 Aggregate가 단일 NestJS app 내 모듈로 공존
- 트랜잭션 경계 = Aggregate 경계 (Vernon Rule 1) 직접 보장
- 부하 테스트(Phase 4·5)가 단일 프로세스 대상으로 단순화

부정:
- 향후 도메인 확장 시(외부 결제·이메일 서비스 등) BC 분리 트리거 발생 가능. 본 ADR은 그 시점에 supersede 후보

중립:
- Phase 3 EDA(ADR-0004) 도입 시 cross-Aggregate 이벤트(notification 모듈이 blog/user 이벤트 구독)가 등장하나 모두 단일 BC 내부 통신
