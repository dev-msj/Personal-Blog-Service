# ADR-0004: Event-Driven Architecture + Publisher-Subscriber

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 3 (TP1)
- Tags: async, eda, publish-subscribe

## Context and Problem Statement

현 시스템은 모든 처리가 동기 트랜잭션 내에서 수행된다 (글 조회 → hits UPDATE 같은 트랜잭션, 좋아요 → 좋아요 카운트 즉시 갱신). 다음 문제 발생:
- 응답 시간이 부가 처리(집계·알림·감사) 부하에 종속 (Forces F1: 응답성 vs 완결성)
- 댓글·답글 작성 시 알림 발송 같은 신규 부가 효과를 추가하면 트랜잭션이 더 무거워짐
- 도메인 이벤트가 명시되지 않아 downstream 확장(검색 인덱스·통계 등) 시 호출 사이트별 hard-coded 분기 필요 (Causes of Redesign 6: Tight coupling)

학습 목표 BP1 "비동기 시스템 적용" — cache/queue/Kafka 모두 경험.

Forces:
- F1 응답성 vs 완결성
- F6 비동기 대상의 자연스러운 존재 (hits 집계·좋아요 카운트·알림 발송)
- 학습 가치 = EDA 패턴 운용 경험

## Considered Options

(1) Event-Driven Architecture + Publisher-Subscriber (선택)
- 구조: Aggregate Command 결과로 domain event 발행. consumer가 구독하여 비동기 처리. Transactional Outbox(ADR-0005)로 발행 보장
- Pros: BP1 학습 목표 충족, 응답 시간 단축 (부가 처리 분리), downstream 확장이 새 consumer 추가만으로 가능, Aggregate 경계 외부는 eventual consistency (Vernon Rule 4 정합)
- Cons: 인프라 복잡도 증가 (Kafka + Outbox + processed_events), at-least-once 의미 + Idempotency 부담

(2) 동기 처리 유지
- Pros: 인프라 단순
- Cons: BP1 학습 목표 미달성

(3) 동기 처리 + hits만 비동기
- Pros: 점진 도입
- Cons: 부분 도입은 이벤트 계약 일관성 상실, 알림·집계 등 추가 확장 시 재설계 비용

(4) CQRS + Event Sourcing
- Pros: 가장 표준적인 EDA 완성형
- Cons: Phase 3 범위 초과 (감수 제약 — 학습 프로젝트 규모 대비 과다), Event Log 저장소 + 재구성 로직 부담

## Decision Outcome

(1) 채택. Phase 3에서 EDA 도입:
- 발행 대상: User Aggregate (UserRegistered/LoggedIn/...) + Post Aggregate (PostCreated/Viewed/Liked/CommentCreated/...) — application-arch.md §Cross-Aggregate 이벤트 인벤토리
- 발행 보장: Transactional Outbox (ADR-0005)
- 브로커: BullMQ (작업 큐) + Kafka (이벤트 스트리밍) 동시 채택 (ADR-0006)
- Idempotency: processed_events 테이블 UNIQUE 제약 (event_id)
- 순서 보장: Aggregate별 파티션 키 (aggregate_type + aggregate_id)
- 의미론: at-least-once + consumer Idempotency = effective exactly-once

## Consequences

긍정:
- 응답 latency 단축 (Phase 4 측정 대상): hits 집계·좋아요 카운트 캐시·알림 발송이 모두 응답 트랜잭션 외부
- 신규 consumer 추가(검색 인덱스·이메일 발송 등) 비용 낮음
- async.md §1.2 혼합 UC 경계 원칙 [확정] 정합: 핵심 트랜잭션 동기 + 부가 효과 비동기

부정:
- 인프라 복잡도: Kafka KRaft + BullMQ Redis + Outbox relay worker + processed_events 운영
- 디버깅 복잡도: cross-process 추적 필요 → Correlation ID 전파 (ADR-0008) + Tempo 트레이싱 (ADR-0009) 의존
- 이벤트 스키마 진화 비용: backward compatibility 정책 운영 (async.md §4.4)

중립:
- Phase 4 부하 테스트: 비동기 경로 latency 별도 시나리오 측정
- observability Extension의 핵심 의존 (Correlation ID 전파 + Consumer lag 메트릭)
