# ADR-0010: Adjacency List for Reply (깊이 1 제한)

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 1
- Tags: data-design, hierarchy, comment

## Context and Problem Statement

Phase 1 댓글/답글 도입(BP3 / TP3 / UC-8·UC-9)에서 트리 구조 표현 방식 선정 필요. 본 프로젝트 도메인 결정: Reply는 Comment에 대해서만 작성 가능, Reply에 대한 Reply는 불가 (깊이 1단계 제한). 깊이 N의 임의 트리가 필요하지 않다.

Forces:
- 도메인 단순성 (Vernon Rule 2: 작은 Aggregate 선호)
- 깊이 1 제한이 도메인 결정 (BP3 학습 범위)
- 쿼리 패턴: Comment 목록 + 각 Comment의 Reply 목록 (보통 N 작음)
- INSERT 패턴: 단순 (Comment → Reply 단방향)

## Considered Options

(1) Adjacency List (`reply.comment_id` 단일 외래키) (선택)
- 구조: `reply (reply_id PK, comment_id FK → comment, user_id, content, ...)`
- Pros: INSERT/쿼리 모두 단순, 깊이 1 도메인에 자연스러움, 스키마 강제 (parent_reply_id 없음 → Reply→Reply 불가)
- Cons: 깊이 N 확장 시 재귀 쿼리 비용 큼 (현재 비해당)

(2) Closure Table
- 구조: `comment_reply_closure (ancestor_id, descendant_id, depth)` 별도 테이블
- Pros: 깊이 N 트리 쿼리 효율
- Cons: 깊이 1 도메인에 과다 (N×depth 행 생성), INSERT 시 추가 행 생성 비용

(3) Nested Sets (Modified Preorder Tree Traversal)
- Pros: 부분 트리 쿼리 효율
- Cons: 삽입/삭제 시 좌우 인덱스 전체 갱신 비용 큼

(4) Materialized Path (경로 문자열 저장)
- 구조: `path VARCHAR("/comment_id/reply_id/...")`
- Pros: 단일 컬럼 조회
- Cons: 깊이 변경 시 모든 자손 path 업데이트 필요, 인덱스 효율 낮음

## Decision Outcome

(1) Adjacency List 채택. Phase 1 스키마 (canonical DDL은 data-design.md §reply primary).

핵심 구조:
- `reply_id BIGINT AUTO_INCREMENT PK`
- `comment_id BIGINT FK → comment.comment_id ON DELETE CASCADE` — 단일 외래키 (parent_reply_id 컬럼 부재)
- `user_id BIGINT FK → user.user_id ON DELETE CASCADE`
- `content VARCHAR(1000)`
- `created_datetime, modified_datetime DATETIME(3)`
- `INDEX (comment_id, created_datetime ASC, reply_id ASC)` — Comment별 시간순 조회 최적화

깊이 1 강제: 스키마로 직접 표현 불가 (parent_reply_id 컬럼 부재로 간접 강제). Reply Service에서 "Reply에 대한 Reply 생성 API 미제공"으로 application-level 강제.

## Consequences

긍정:
- 도메인 단순성 일치 (깊이 1 = Adjacency List 최적)
- INSERT 1 row / 쿼리 단순 `WHERE comment_id = ?`
- Comment 삭제 시 Reply CASCADE 자연 (Post Aggregate Invariant)

부정:
- 깊이 N 확장 트리거 발생 시 Closure Table 마이그레이션 필요
- application-level 강제 (스키마 강제 불가)

중립:
- Phase 1 Aggregate Invariant: "Reply의 깊이는 1단계로 제한"
- application-arch.md §Post Aggregate Vernon Rule 2 적용 판단: Comment/Reply를 Post 내부 Entity로 유지 (현 단계)
