# ADR-0007: Idempotency Key Pattern (API 수신 측)

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 1
- Tags: api, reliability, idempotency, security

## Context and Problem Statement

HTTP 클라이언트 네트워크 재시도 시 같은 write 요청이 중복 도달할 수 있다. 좋아요 토글 / 댓글 작성 같은 상태 변경 연산에서 중복 처리는 데이터 오염을 야기한다. 클라이언트 디바운싱은 네트워크 재시도(타임아웃 + retry)에 무력하며, DB UNIQUE 제약만으로는 상태 변경(취소 후 재실행)의 멱등성을 보장할 수 없다.

Forces:
- 클라이언트 재시도는 정당한 통신 패턴
- 학습 가치 = Stripe/GitHub 업계 표준 패턴 운용 경험 (BP3)
- 이벤트 수신 측 Idempotency(async §6)와 영역 분리 — API 수신 측은 HTTP 계층의 멱등성

## Considered Options

(1) 클라이언트 발급 `Idempotency-Key` 헤더 + Redis 저장 (선택)
- 헤더: `Idempotency-Key: <UUID v4>` (RFC 4122)
- 저장: Redis `idempotency:{user_id}:{key}` TTL 24h, value = `{statusCode, responseBody, method, path, processedAt}`
- Pros: Stripe/GitHub 업계 표준, 응답 재반환 가능 (네트워크 재시도 UX 우수), 클라이언트 발급이라 새 ID 발급 문제 없음
- Cons: Redis 의존, 클라이언트가 UUID 발급 책임

(2) 무상태 중복 방지 (클라이언트 디바운싱)
- Pros: 서버 변경 없음
- Cons: 네트워크 재시도에 무력, 멀티탭/멀티디바이스 대응 불가

(3) DB UNIQUE 제약만
- Pros: 가장 단순
- Cons: 좋아요 취소 후 재좋아요 같은 상태 변경 모호 케이스 처리 어려움

(4) 서버 생성 트랜잭션 ID
- Pros: 서버 통제
- Cons: 클라이언트 재시도 시 새 ID 발급되어 중복 방지 실패 (Idempotency-Key 본질 위반)

## Decision Outcome

(1) 채택. Phase 1 도입:

대상 엔드포인트 (필수):
- 모든 Write API (POST/PATCH/DELETE) — 특히 `/posts/:postId/likes`, `/posts`, `/posts/comments`, `/posts/comments/:commentId/replies`
- 읽기 API (GET): 적용 제외 (GET은 idempotent이어야 하며 클라이언트 재시도 방어 불필요)

중복 감지 처리:
1. `user_id + idempotency_key` 조합으로 Redis GET
2. 값 존재 + 완료 → 원본 응답 재반환 (status, body 복원)
3. 값 존재 + pending → 409 Conflict + Retry-After: 5
4. 값 없음 → SETNX로 pending 저장 후 처리, 응답 후 완료로 UPDATE

저장 정책: TTL 24h. Helland ACM Queue 2012 + Stripe 권장.

## Consequences

긍정:
- 네트워크 재시도 안전 (응답 재반환으로 UX 일관성)
- 좋아요 토글 같은 상태 변경 연산도 멱등 보장
- DT-1 (Idempotency-Key Decision Table) 본체 제공

부정:
- Redis 의존 (이미 캐시·Rate Limit·BullMQ에서 사용 중이라 추가 비용 낮음)
- 클라이언트 SDK가 UUID v4 발급 책임 — API 문서에 명시 필요
- Phase 4 부하 테스트: Idempotency 검증 자체의 latency 영향 측정 대상
