# ADR-0002: JWT access + refresh 이중 검증 (서버측 DB 대조 + Token Rotation)

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 0 (baseline 명문화), Phase 5 (해시 저장 검토)
- Tags: authentication, jwt, session, security

## Context and Problem Statement

전통적 표준 JWT 모델은 stateless하여 서명·만료만으로 통과시킨다. 이 경우 서버측 세션 즉시 무효화가 불가능하며 탈취된 토큰은 최대 수명(보통 1시간)까지 유효하다. 본 프로젝트는 학습 환경이지만 로그아웃·세션 무효화·탈취 대응 같은 보안 시나리오를 운영 가능한 구조로 만드는 것이 학습 목표다.

Forces:
- 즉시 무효화 가능성 vs JWT의 stateless 이점
- 학습 가치: 표준 JWT 한계와 보완 방식의 트레이드오프 경험
- UC-4 Invariant "Rotation 원자성" (access + refresh 동시 재발급 + DB 저장값 함께 갱신)
- OWASP JWT Cheat Sheet "Short-lived access tokens + rotating refresh tokens"

## Considered Options

(1) JWT access(1h) + refresh(30d) 이중 검증 + 서버측 DB 저장 refresh + Rotation (선택)
- Pros: 즉시 무효화 가능 (DB의 refresh 값 변경/삭제), Rotation으로 탈취 시 detection 가능 (이전 token 재사용 시 401), Stateless의 trade-off를 학습 가치로 전환
- Cons: refresh 시 DB 조회 1회 추가, refresh token 평문 DB 저장 (Phase 5 해시 저장 검토 ADR 별도)

(2) AccessToken 단일 (장기 수명)
- Pros: 가장 단순, DB 의존 없음
- Cons: 즉시 무효화 불가, 탈취 시 최대 수명까지 노출 (docs/tech-notes/token-validation-strategies/ 분석 결론)

(3) Server-side 세션 (전통적 sticky session)
- Pros: 가장 안전한 세션 관리
- Cons: NestJS REST API 컨벤션과 거리, 학습 목표(JWT 운용 경험)와 직교

(4) Refresh Token Reuse Detection (사용한 refresh 재사용 감지 시 모든 세션 무효화)
- Pros: 탈취 시 자동 lockout
- Cons: Phase 2 관측성 도입 후 검토 항목으로 분리 (현 단계 미적용 — 후속 ADR 후보)

## Decision Outcome

(1) 채택. 모든 보호 엔드포인트는 두 토큰 모두 유효해야 통과:
- AccessToken (Bearer header) 서명/만료 검증
- RefreshToken (HTTPOnly cookie) DB 저장값과 대조
- Refresh 시: access + refresh 모두 재발급 + DB 원자적 갱신 (QueryRunner 트랜잭션)

수명: access 1h / refresh 30d. 클라이언트는 access 만료 시 `POST /users/auth/refresh` 호출.

## Consequences

긍정:
- 즉시 무효화 가능 (refresh DB 삭제 → 401)
- Rotation으로 토큰 이중 사용 detection 가능
- security.md §1 토큰 수명·회전 [확정] 결정과 정합

부정:
- 매 요청마다 DB 조회 (refresh 검증) — Phase 4 부하 테스트에서 영향 측정 대상
- refresh 평문 DB 저장 (Phase 5에 해시 저장 전환 검토 [가이드])

중립:
- 라이브러리: `jsonwebtoken` 직접 사용 (passport 미사용). NestJS 표준과 다른 선택은 의도된 학습 산출물(블로그 글감 — Phase 0 Type A 완료)
- Phase 5 RFC 9457 전환(ADR-0014) 후에도 본 이중 검증 구조 유지
