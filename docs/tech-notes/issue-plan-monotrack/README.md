# 다음 작업 이슈를 매번 분석하지 않게 만들기 — issue-plan을 모노트랙으로

> Type B (회고 블로그) / Phase 0 회고
> 관련 산출물: MCPSI 파이프라인, plan-manager 스킬, issue-plan.md

## 개요

이 글은 AI 에이전트(Claude Code)와 함께 개발 워크플로우를 운영하면서, 사람용으로 만들어진 멀티트랙 이슈 관리 방식이 AI 협업에서 어떻게 비용을 만드는지 관찰하고, 다음 작업 후보를 정적으로 결정하는 모노트랙 방식으로 전환한 과정을 정리한다.

핵심 메시지는 한 줄이다. AI가 사람보다 빠른 환경에서는, 매 PR마다 다음 작업을 "분석해 판단"하게 두는 것 자체가 비용이다. 분석 결과를 한 곳에 정적으로 모아두고 매번은 그 결과만 읽게 하면, AI의 속도 이점이 살아난다.

본문은 시간순으로 간다. 시작점(MCPSI) → 관찰 → 다섯 가지 문제 → 발상 전환 → 결정 → 가정과 진화 경로 순서다.

## 시작점 — MCPSI와 issue-plan.md

Phase 0 진입 전, MCPSI라는 소프트웨어 설계 절차를 먼저 만들었다. Meeting Log → Context → Problem → Solution → Implementation의 5단계로, 요구사항 분석부터 구현 이슈 생성까지를 문서화한다.

설계 절차를 둔 이유는 한 가지다. 동일 프로젝트를 AI 에이전트를 바꿔 가며 구현하든, 같은 AI로 여러 번 구현하든, **항상 일관된 로직과 80% 이상 동일한 품질의 코드를 생산할 수 있어야 한다.** 비개발자가 참여하는 협업에서도 같은 보장이 필요했다.

이 목표를 위해 설계 결정과 이슈 의존성을 사전에 문서화하고, 그 결과 산출물 중 하나로 `issue-plan.md`가 있다. 이슈를 어떤 순서로, 어떤 의존성 아래에서 구현할지 정의한 문서다.

워크플로우 스킬은 이슈 구현과 리뷰가 끝나면 다음 작업할 이슈를 PR 코멘트로 알려 주도록 했다. 의도는 명확했다. **작업자가 개발 내용을 몰라도, 코멘트만 보고 워크플로우 스킬만 호출하면 다음 작업으로 진입할 수 있게 한다.**

여기까지는 합리적으로 보였다.

## 관찰 — AI가 매 PR마다 다음 작업을 분석한다

실제 운영하면서 관찰한 동작은 의도와 달랐다.

`issue-plan.md`는 멀티트랙으로 운영되고 있었다. 핵심 기능 구현 이슈가 있고, 별도 섹션에 "신규 등록 이슈"가 쌓인다. 신규 등록 이슈가 따로 존재하는 이유는 다음 분류 때문이다.

리뷰 사이클에서 나온 지적 사항은 네 가지로 처리된다.

- 즉시 해결
- TODO 주석으로 기록
- 신규 이슈 등록
- 거부

이 중 TODO 기록과 신규 이슈가 분리되어 있는 이유는, 현재 Phase 범위를 벗어나는 지적은 TODO만 남기게 해서 이슈 목록이 오염되는 것을 막기 위해서다. 그럼에도 발생하는 신규 이슈들은 기존 이슈와의 의존성이 있을 수도, 없을 수도 있다. 의존이 있다면 강도도 다양하다.

이 구조 위에서 클로드 코드는 PR을 머지하기 전마다 `issue-plan.md` 전체를 분석해, 다음 작업 가능한 이슈를 판단해 알려 줘야 했다. 그리고 판단 결과는 거의 항상 같은 패턴이었다.

**핵심 기능을 우선으로 작업하도록 제시 + 신규 등록 이슈 중 작업 가능한 것을 서브로 제안**.

## 멀티트랙 제안 방식의 다섯 가지 문제

이 제안 방식은 다섯 가지 문제를 만들었다.

| 번호 | 문제 |
|---|---|
| 1 | 비개발자는 제안만 보고 다음 작업을 스스로 판단하기 어렵다 |
| 2 | 클로드 코드가 매번 `issue-plan.md`를 분석하고 판단해야 한다 |
| 3 | 핵심 기능 우선이 항상 최적은 아니다 |
| 4 | 의존 강도가 높은 이슈를 미루면 사이드 이펙트가 누적된다 |
| 5 | 의존 강도가 낮은 이슈는 영원히 후순위로 밀린다 |

문제 3을 풀어 쓰면 이렇다. "핵심 기능 우선 구현"은 빠른 비즈니스 가치 검증이 최우선인 환경에서는 합리적이다. 그러나 이 프로젝트는 개인 프로젝트이고, Phase 단위 구현 계획이 이미 확정된 상태다. 그리고 AI의 작업 속도는 사람보다 매우 빠르다. 이 조건에서는 신규 등록 이슈까지 함께 처리하며 이슈 목록을 클린하게 유지하는 편이, 리뷰 지적 누적을 줄이고 결과적으로 빠르다.

문제 4와 5는 멀티트랙 제안 방식의 구조적 약점이다. 핵심 기능을 먼저 처리하면 의존 강도가 높은 신규 이슈는 사이드 이펙트가 미리 대비되지 않은 상태에서 만나게 된다. 리뷰가 지적할 거리가 늘고, 신규 이슈가 또 등록된다. 반대로 의존 강도가 낮거나 없는 신규 이슈는 핵심 기능이 끝날 때까지 영원히 후순위에 머문다. 어느 쪽이든 누적 부담은 결국 사람이 다시 정리해야 할 빚으로 돌아온다.

문제 2는 이 모든 비용의 직접적 원인이기도 하다. 매 PR마다 클로드 코드가 `issue-plan.md` 전체를 다시 읽고, 의존성을 재해석하고, 가용 후보를 골라 우선순위를 매긴다. 같은 분석을 매 PR마다 새로 한다. AI가 사람보다 빠른 속도를 만드는 환경인데, 그 속도의 일부를 매번 같은 분석에 쓰고 있었다.

## 발상 전환 — 단일 작업 후보 + 책임 분리

문제의 본질은 "다음 작업 후보 목록을 매번 분석한다"는 데 있었다. 발상을 바꾸면 다음 두 줄이 된다.

- 매번 분석하지 않는다.
- 분석 결과를 정적으로 한 곳에 저장하고, 매번은 그 결과만 읽는다.

이렇게 두면 워크플로우 단계마다 클로드 코드가 해야 할 일이 줄어든다. PR 머지 전에 결정해야 하는 것은 "다음 작업 후보가 무엇인가"가 아니라 "다음 작업 후보를 읽고 작업으로 진입한다"뿐이다. 분석은 분석을 책임지는 다른 주체가 따로 수행한다.

이 책임 분리가 자연스럽게 단일 작업 후보 제안이라는 방식으로 이어진다. 후보가 여러 개면 결국 매번 우선순위를 다시 결정해야 한다. 후보가 정확히 하나면 더는 결정할 것이 없다.

## 모노트랙 결정과 plan-manager의 신설

이렇게 해서 `issue-plan.md`를 **모노트랙**으로 운영하기로 결정했다. 작업 진입 경로가 하나, 다음 작업 후보는 항상 정확히 하나.

분석 책임을 가져갈 주체로 `plan-manager` 스킬을 신설했다. 책임은 두 가지다.

- 초기 이슈 생성 시 의존성을 분석해 `issue-plan.md`를 작성한다.
- 신규 이슈가 등록될 때만 의존성을 재분석하고 plan을 갱신한다.

분석은 "초기 1회 + 신규 이슈 등록 시"로 한정된다. 그 외에는 분석이 일어나지 않는다. PR 머지 전 라우팅에 필요한 것은 plan을 읽어 다음 후보를 꺼내는 단순 조회뿐이다.

작업 경로별로 작업자 정보도 plan에 적어 두었다. 작업자가 여러 명이더라도, 문서 변경 시 plan-manager가 어떤 작업 경로의 내용을 수정해야 하는지 즉시 알 수 있다. 이 점은 다음 단계의 작업자 분할에서 다시 활용된다.

이 결정의 근거를 정리하면 세 가지다.

1. 분석 비용 제거. AI가 매번 같은 분석을 반복하지 않게 된다. 작업 진입이 즉시 가능하다.
2. 비개발자 친화. 코멘트의 단일 후보만 보고 워크플로우 스킬을 호출하면 된다. 비개발자가 우선순위 판단에 개입할 일이 없다.
3. 병렬화의 기반. 모노트랙이 검증되어야 그 위에서 병렬 처리로 안전하게 확장할 수 있다 (다음 절 참조).

## 다음 단계 — 작업자 분할과 워크플로우 간소화

모노트랙은 이 결정의 종착점이 아니라 시작점이다. 결정 시점에 함께 설계한 확장 한 가지와, 모노트랙이 검증되어야 진입할 수 있는 후속 단계가 따로 있다.

### 작업자 분할 — 모노트랙과 함께 설계된 확장

작업자가 여러 명이 될 때를 대비해, 결정 시점에 다음 확장을 함께 설계해 두었다.

- 작업 진입 경로를 작업자 수만큼 분할한다.
- 각 진입 경로는 여전히 모노트랙이다.
- plan-manager는 어떤 진입 경로가 어떤 작업자에게 속하는지 알고 있다.

이 구조는 모노트랙의 이점을 유지하면서 작업자 N명을 수용한다. 작업자 정보를 plan에 적어 둔 것이 이 확장의 직접적 기반이다.

핵심은 **병렬화를 "후보를 동적으로 분배"가 아니라 "정적으로 분할된 트랙들의 병렬 진입"으로 푼다**는 점이다. 동적 분배는 매번 작업자 가용 상태를 보고 후보를 재할당해야 하지만, 정적 분할은 각 트랙이 독립적으로 진행된다. 모노트랙의 핵심 이점(분석 비용 제거)이 그대로 보존된다.

### 워크플로우 간소화 — 검증해야 할 다음 과제

현재 워크플로우 스킬은 MCPSI 설계 문서가 없던 시절에 만들어졌다. 그래서 검증과 리뷰 절차가 과도하다.

워크플로우 1회당 소요 시간과 자원을 측정한 결과는 다음과 같다.

- 시간: 1시간 내외
- Claude Max 5x 요금제 기준, 5시간 세션 사용량의 약 25%

이 비용으로는 세션 안에서 이슈를 병렬 처리하기 어렵다. 워크플로우 간소화가 선행되어야 세션 단위 병렬화가 현실화된다. MCPSI 설계 문서가 충분한 근거 역할을 수행하는지 다음 작업을 진행하며 검증하고, 충족된다면 워크플로우의 검증/리뷰 일부를 축소한다. 간소화가 완료되면 세션 사용량을 다시 측정해 병렬 처리 도입 여부를 결정한다.

### 도달하려는 모습

두 단계를 모두 적용하면 다음 그림이 된다.

- 작업자 N명, 각자의 진입 경로(트랙)는 모노트랙으로 정적 분할
- 워크플로우는 간소화되어 1회 비용이 작음
- 사람이 관여하지 않아도, 설계 문서만 있으면 모든 이슈가 자동 처리

설계 문서를 작성하는 것까지가 사람의 일이고, 그 이후는 자동화의 영역이 된다. 이것이 모노트랙 결정이 향하는 그림이다.

## 마무리

AI 에이전트 협업의 이슈 관리는 사람용 패턴을 그대로 옮기면 비용을 만든다. "다음 작업 후보를 매번 분석한다"는 사람의 익숙한 절차가, AI의 속도 이점을 깎아 먹는 지점이 된다.

분석을 정적으로 한 번만 수행하고, 매 작업 진입은 그 결과만 읽게 한다. 이 결정 한 줄이 모노트랙의 본질이다. 그 위에서 작업자 분할이 함께 설계되었고, 워크플로우 간소화가 다음 검증 과제로 남는다.

본 결정은 현재 단일 작업자 환경에서 운영 중이며, 다음 단계는 워크플로우 간소화의 측정과 검증이다.
