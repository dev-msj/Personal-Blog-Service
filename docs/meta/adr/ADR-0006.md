# ADR-0006: BullMQ + Kafka 동시 채택 (작업 큐 + 이벤트 스트리밍 분리)

- Status: Accepted
- Date: 2026-05-12
- Phase: Phase 3
- Tags: async, broker, queue, kafka

## Context and Problem Statement

ADR-0004 EDA 도입 시 메시지 브로커 선정 필요. 본 프로젝트 비동기 처리는 두 가지 성격이 혼재한다:
- 작업 큐 패러다임: 즉시 실행 / 1회 처리 / Rate Limiting 필요한 백그라운드 작업 (알림 발송, 이메일 발송, 외부 API 래퍼)
- 이벤트 스트리밍 패러다임: 도메인 이벤트 / 여러 consumer 구독 / 장기 보존 / 순서 보장 (PostViewed, PostLiked, CommentCreated 등 cross-Aggregate 이벤트)

학습 목표 BP1 "cache, queue, redis, kafka" 모두 경험 — 두 패러다임을 분리 학습 가치.

Forces:
- 단일 브로커 운용 단순성 vs 패러다임 분리 학습 가치
- context.md [기술 스택 제약] 기존 Redis 스택 활용 (BullMQ는 Redis Streams 기반)
- 학습 프로젝트 인프라 복잡도 한계

## Considered Options

(1) BullMQ + Kafka 동시 채택 (선택)
- BullMQ (`@nestjs/bullmq` 공식): 작업 큐. notification 큐, email 큐
- Kafka (KRaft 모드): 이벤트 스트리밍. user.events / post.events 토픽
- Pros: 두 패러다임 분리 학습 (BP1 충족), 기능 적합도 (작업 큐 = 우선순위/지연/Rate Limit, 이벤트 = 순서/보존/replay)
- Cons: 인프라 컨테이너 2종 추가 (Redis 기존 + Kafka 신규), 운영 학습 부담 2배

(2) Bull (legacy `@nestjs/bull`)
- Pros: 단순
- Cons: 유지보수 모드, 공식 문서에서 신규 프로젝트 BullMQ 권장

(3) Kafka only
- Pros: 단일 브로커
- Cons: 작업 큐 전용 기능(우선순위/반복/Rate Limit) 부재로 단순 백그라운드 작업에 과도한 인프라, BullMQ 학습 누락

(4) BullMQ only
- Pros: 단일 브로커, 기존 Redis 활용
- Cons: Kafka 학습 목표 미달, 도메인 이벤트의 장기 보존·replay 어려움

(5) RabbitMQ
- Pros: 작업 큐와 pub-sub 모두 적합
- Cons: Redis 이미 있는데 추가 브로커 → 인프라 중복. 작업 큐 편의성은 BullMQ 우위

(6) AWS SQS / Agenda / Redis Pub/Sub 직접 구현
- 각각 클라우드 의존(out-of-scope) / MongoDB 의존(스택 불일치) / Job lifecycle 자체 구현 부담(바퀴 재발명)

## Decision Outcome

(1) 채택. 용도별 분리:

| 브로커 | 용도 | 큐/토픽 |
|---|---|---|
| BullMQ | 작업 큐 (1회 처리, Rate Limiting) | `notification`, `email`(향후) |
| Kafka KRaft | 도메인 이벤트 (다중 consumer, 순서 보장) | `user.events`, `post.events`, `<topic>.DLQ` |

토픽 분리: Aggregate 경계 기반 (Kleppmann Ch.11 "Topic per entity type"). 파티션 수 = 토픽당 3개 (학습 목표 — 병렬 처리 + 순서 관찰).

Consumer 그룹 명명: `<service>.<purpose>` (예: `blog-service.hits-aggregator`, `blog-service.notification-publisher`).

## Consequences

긍정:
- BP1 학습 목표 충족 (cache, queue, kafka 3종 운용)
- 패러다임별 자연스러운 도구 선택 (Rate Limit 필요 = BullMQ, 다중 구독 필요 = Kafka)
- 클라우드 배포 트리거 시 두 브로커 모두 관리형 옵션 존재 (AWS MSK + ElastiCache, GCP Pub/Sub + Memorystore 등)

부정:
- docker-compose.yaml에 Kafka KRaft 컨테이너 추가 (Phase 3.3 진입 시점)
- Consumer lag 모니터링이 BullMQ + Kafka 2곳 분리
- DLQ 정책 2가지 (Kafka `<topic>.DLQ` 토픽 + BullMQ 내장 Failed Jobs)

중립:
- BullMQ → Kafka로 작업 enqueue 흐름 가능 (notification-publisher consumer가 CommentCreated 소비 → BullMQ enqueue)
- Phase 4 부하 테스트: 두 브로커 각각 별도 시나리오
